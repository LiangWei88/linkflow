# 编辑器技术选型分析

**版本**: 1.1
**日期**: 2026-03-22
**状态**: 技术决策

---

## 1. 背景

本项目需要实现一个大纲笔记编辑器，核心需求包括：

- 每个 block 独立编辑（点击哪个编辑哪个）
- 无感编辑体验（编辑和预览状态完全一致）
- 键盘导航（方向键在 block 间移动）
- Markdown 基础语法支持
- 双向链接 `[[wikilink]]` 支持

当前使用 Tiptap 编辑器，遇到了以下问题：

1. 点击定位光标不准确
2. 键盘边界导航实现复杂
3. 多行内容需要额外转换
4. 编辑器默认样式与"无感编辑"理念冲突

---

## 2. 方案对比

### 2.1 Tiptap 方案

#### 优势

| 优势 | 说明 |
|------|------|
| Markdown 支持 | 内置 Markdown 解析和渲染，支持粗体、斜体、列表等 |
| 扩展系统 | 可以通过 Extension 扩展功能，如自定义 Mark |
| 富文本能力 | 支持复杂的富文本操作（撤销、格式化等） |
| 社区支持 | 活跃的社区和丰富的文档 |
| 所见即所得 | 实时渲染 Markdown 效果 |

#### 劣势

| 劣势 | 说明 |
|------|------|
| 复杂度高 | 每个功能都需要理解 ProseMirror 底层 |
| 光标控制难 | 点击定位、边界导航需要深度定制 |
| 性能开销 | 每个 block 创建编辑器实例有开销 |
| 样式冲突 | 默认样式与"无感编辑"理念冲突 |
| 学习曲线 | 需要学习 Tiptap 和 ProseMirror API |

### 2.2 原生开发方案

使用 `contenteditable` + 自定义逻辑实现。

#### 优势

| 优势 | 说明 |
|------|------|
| 完全控制 | 光标、选区、键盘行为完全可控 |
| 简单直接 | 每个 block 就是一个 div，逻辑清晰 |
| 性能好 | 无框架开销，轻量级 |
| 无感编辑 | 天然支持编辑和预览状态一致 |
| 易于调试 | 代码逻辑透明，问题容易定位 |

#### 劣势

| 劣势 | 说明 |
|------|------|
| Markdown 解析 | 需要自己实现 Markdown 解析和渲染 |
| 富文本功能 | 撤销、格式化等需要自己实现 |
| 浏览器差异 | 不同浏览器的 contenteditable 行为有差异 |
| 双向链接 | `[[wikilink]]` 需要完全自定义 |

## 4. 决策

### 4.1 最终选择：**原生开发方案**

### 4.2 决策理由

1. **核心需求匹配度高**
   - 项目的核心理念是"无感编辑"，原生方案天然支持
   - 每个 block 独立编辑的场景，原生方案更简单直接

2. **问题解决成本低**
   - 当前 Tiptap 遇到的问题（光标定位、键盘导航）在原生方案中不存在
   - 不需要与框架"对抗"来实现需求

3. **Markdown 需求简单**
   - 项目只需要基础 Markdown（粗体、斜体、链接）
   - 可以使用轻量级 Markdown 解析库（如 marked.js）实现

4. **双向链接都需要自定义**
   - 无论 Tiptap 还是原生，`[[wikilink]]` 都需要自定义实现
   - Tiptap 的 Mark 扩展并不比原生实现更简单

5. **长期维护性**
   - 原生代码逻辑透明，易于理解和维护
   - 不依赖第三方框架的版本更新

## 5. 架构设计

### 5.1 组件分层

```
┌─────────────────────────────────────────────────────────────┐
│                        Outline                               │
│  - 管理全局状态（blocks、editingBlockId、cursorState）        │
│  - 协调 block 间的导航                                       │
│  - 处理树形结构渲染                                          │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                      OutlineItem                             │
│  - 展示单个 block（预览/编辑状态）                            │
│  - 处理点击事件，计算点击位置                                 │
│  - 管理折叠/展开                                             │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                      BlockEditor                             │
│  - 封装 contenteditable 编辑逻辑                             │
│  - 处理输入、光标、键盘事件                                   │
│  - 防抖保存内容                                              │
└─────────────────────────────────────────────────────────────┘
```

### 5.2 为什么保留 OutlineItem 和 BlockEditor 分离？

| 原因 | 说明 |
|------|------|
| **性能优化** | 只为当前编辑的 block 创建编辑器实例，避免为每个 block 创建 |
| **职责分离** | OutlineItem 负责"展示和交互"，BlockEditor 负责"编辑逻辑" |
| **复用性** | BlockEditor 可以在其他地方复用（如独立编辑弹窗） |
| **状态隔离** | 编辑状态只在 BlockEditor 内部管理，不影响其他 block |

### 5.3 为什么用 contenteditable div 而不是 input/textarea？

| 元素 | 支持多行 | 光标控制 | 富文本 | 结论 |
|------|----------|----------|--------|------|
| `<input>` | ❌ 单行 | 有限 | ❌ | 不适用 |
| `<textarea>` | ✅ | 有限 | ❌ | 不适用 |
| `contenteditable` | ✅ | 灵活 | ✅ | **最佳选择** |

`contenteditable` 的优势：
- 支持多行内容
- 光标控制 API（Selection、Range）灵活
- 未来可扩展富文本功能

### 5.4 光标状态管理

```
┌─────────────────────────────────────────────────────────────┐
│                    光标状态（Outline 管理）                   │
├─────────────────────────────────────────────────────────────┤
│ cursorOffset   │ 点击时的文本偏移量                          │
│ cursorPosition │ 导航时的目标位置（start/end/firstLine/lastLine）│
│ savedCursorX   │ 跨 block 导航时保存的水平位置               │
└─────────────────────────────────────────────────────────────┘
```

**状态优先级**：
1. 点击触发：使用 `cursorOffset`
2. 导航触发：使用 `cursorPosition` + `savedCursorX`
3. 默认：光标在开头

### 5.5 键盘导航逻辑封装

将键盘导航逻辑封装到 `useCursorNavigation` hook 中：

```typescript
// src/hooks/useCursorNavigation.ts
export function useCursorNavigation(options) {
  return {
    isCursorAtStart,      // 检测光标是否在开头
    isCursorAtEnd,        // 检测光标是否在结尾
    isCursorAtFirstLine,  // 检测光标是否在第一行
    isCursorAtLastLine,   // 检测光标是否在最后一行
    handleKeyNavigation,  // 处理键盘导航事件
  }
}
```

**好处**：
- 逻辑复用
- 便于单元测试
- 后续可扩展其他快捷键

## 6. 键盘导航行为规范

### 6.1 方向键行为

| 按键 | 条件 | 行为 |
|------|------|------|
| ← | 光标在 block 开头 | 跳到上一个 block 的结尾 |
| → | 光标在 block 结尾 | 跳到下一个 block 的开头 |
| ↑ | 光标在第一行（或单行） | 跳到上一个 block 的最后一行，保持水平位置 |
| ↓ | 光标在最后一行（或单行） | 跳到下一个 block 的第一行，保持水平位置 |

### 6.2 光标位置保持

跨 block 导航时，需要保持光标的水平位置：
1. 记录当前光标的 X 坐标
2. 在目标 block 中找到最接近该 X 坐标的位置
3. 设置光标到该位置
